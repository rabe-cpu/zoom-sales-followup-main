# 04 Email Generation Rules

## 基本構成

1. 件名
2. 宛名
3. 挨拶・名乗り
4. お礼
5. 商談で話した内容の回収
6. 判断軸の整理
7. 資料・プラン・シミュレーション案内
8. 参考動画
9. ネクストアクション
10. 資料送付
11. 安心導線
12. 季節の結び
13. 締め

## 件名（固定）

顧客送付用メール、Gmail下書き、社内確認用の4ソーシャルスタイル別全文メール案の件名は、全て下記に統一する。
顧客名、日付、担当者名、商談内容、スラッシュ区切りの追記などで変更しない。

```text
【株式会社NEXT】個別相談会の御礼(総合物販システム_アクセス)
```

## 冒頭の会話反映

冒頭は、次の3つを入れる。

- 顧客が話した背景
- 顧客が気にしていた判断軸
- 営業が受け止めた自然な一文

例:

`本日お話を伺う中で、上田さまは新しいことをただ増やしたいというより、飲食店やキッチンカーで今ある人員・空き時間をうまく使えるかを、かなり現実的に見られているんだなと感じました。`

## 参考動画の書き方

自然なのは、URLの前後に短く説明を入れる形。

構成:

1. URL前: なぜこの動画を選んだか
2. URL
3. URL後: どこを見ると判断材料になるか

URL後の説明を省くと、営業がちゃんと選んだ感じが弱くなる。

ただし、URL後を長く書くと宣伝っぽくなるため1文で止める。

## ソーシャルスタイル別全文メール案（社内確認用）

社内確認用ドキュメントには、`knowledge/12_social_style_email_variants.md` に従い、4つのソーシャルスタイル別全文メール案を追加する。

重要:

- 営業後メール生成時は `knowledge/benchmark_playbooks/suzue_benchmark.md` を必ず参照し、`benchmarkCoach`、`winningPatterns`、`phasePlaybooks`、`customerAttributePlaybooks` に反映する。
- OpenAI Vector Storeが利用可能な場合は、`knowledge/rag/suzue_vector_store.md` に従い、メール本文を書く前に今回商談の論点で鈴江商談RAGを検索する。
- RAG検索結果は、鈴江ベンチマークの静的ナレッジより粒度の細かい「似た場面の営業型」として使い、`benchmarkCoach`、`winningPatterns`、`phasePlaybooks`、`customerAttributePlaybooks`、価格質問対応、不安が出た時の戻し方、クロージングに反映する。
- 鈴江ベンチマークは文字起こしベースの営業型として使い、顧客事実の根拠にはしない。
- RAG検索結果も顧客事実の根拠にはしない。今回顧客が話していない背景、金額、感情、成果期待を足さない。
- 顧客送付用メールと社内確認用docxには、Vector Store検索ログ、鈴江商談名、原文引用を出さない。出すのはトップ営業の指導文に変換した内容だけにする。
- 音声・映像コーチング、声色・話速評価、録音練習、模写練習メニューは出力しない。
- 顧客の会話からソーシャルスタイルを判定しない。
- `この顧客はDriver`、`Analytical型と推定` のように断定しない。
- 同じ商談事実を、Driver / Driving、Analytical、Amiable、Expressive の4つの伝え方に変換する。
- 各スタイルは差し替え段落ではなく、件名から署名・固定資料URL・固定フォームURLまで含む全文メール案として作る。
- 顧客送付用メールには、4スタイル別案や社内フィードバックを混ぜない。
- 4スタイル別案は、営業担当が送信前に表現を選ぶための社内確認用として残す。
- 各スタイルには、全文メール案、顧客反応シグナル、営業担当が選ぶ目安、次回商談での質問例、そのまま使える返答例、価格・費用質問への返し方、避ける言い方、伝え方メモ、次の一手、ベンチマーク営業トーク、ベンチマーク営業台本、文脈接続メモを含める。
- 4スタイル別メール案の後ろに「この文面が効く理由」は出さない。理由説明の本文も独立項目にせず、必要な背景は「営業担当が選ぶ目安」「伝え方メモ」「文脈接続メモ」に短く統合する。
- 商談フィードバック要素として、overallSummary / customerInsights / cognitiveBias / expectationGap / strengths / improvementPoints / coachingCards / winningPatterns / stageStrategy / phasePlaybooks / customerSignals / temperature / cues / effectiveQuestions / effectiveReplies / priceQuestionHandling / avoidedTalk / delivery / nextBestAction / benchmarkTalk / benchmarkCoach / contextBridge / customerAttributePlaybooks / riskAlerts を社内確認用に反映する。
- 価格・費用質問への返し方では、価格だけを即答して終わらせず、目的、作業時間、予算感、導入時期、意思決定者、不安の種類を確認し、合うプランや判断条件へ戻す。
- 4スタイルの違いは、語尾や文量だけで出さない。Driverは結論・主導権・次アクション、Analyticalは根拠・条件・例外、Amiableは安心・合意形成・相談導線、Expressiveは未来像・承認・ストーリーで分ける。
- 各スタイルに、価格質問対応、不安が出た時の戻し方、クロージング、ストレス反応へのリカバリーを入れる。
- 社内確認用docxに、`overallSummary`、`hiddenNeeds`、`name=`、`item=`、`theme=`、`benchmarkCoach.script` などの内部キー名を表示しない。内部では構造化してよいが、出力は必ず日本語見出しと自然文に変換する。
- 商談フィードバックは、トップ営業マンが目の前で営業担当に指導している口調にする。箇条書きの羅列ではなく、「まずここを見ます」「次回はこう聞きます」「この場面ではこう返します」のような実践指導文を中心にする。
- 箇条書きは、次回質問や確認項目など、営業がそのままチェックする用途に限定する。分析項目の羅列だけで終わらせない。

## 商談フィードバック抽出の順序

メール本文を書く前に、内部で以下の順に商談フィードバックを整理する。

1. 総合概要を作る。
   - 商談の全体要約、顧客の状況、検討背景、判断軸、現在の進み具合を300〜500文字程度でまとめる。
   - トップ営業・勝ち商談と比べた差分があれば、差分と不足行動を文章で整理する。数値スコアは出さない。
2. 顧客インサイトを作る。
   - 潜在ニーズ、言い切っていない不安、比較対象、判断基準、購買温度感を整理する。
3. 認知バイアスと期待値のズレを作る。
   - 認知バイアスは name / description / counterMeasure。
   - 期待値のズレは topic / gap / solution。
4. 良かった点と改善ポイントを作る。
   - それぞれ item / detail の形で3〜5個に絞る。
   - 改善ポイントは、次にどう言い換えるかまで分かる内容にする。
5. AIコーチングカードを4枚作る。
   - theme / scene / insight / issue / strategy / script / outcome を必ず入れる。
   - script は営業担当がそのまま話せる2〜4文にする。
   - docx表示では `theme=` などのキーを出さず、「この場面では」「トップ営業なら」「そのまま使える台詞」の自然な指導文に変換する。
6. 再現する勝ち筋を作る。
   - 勝ち筋名 / 使う場面 / 顧客シグナル / トップ営業の動き / コピートーク / なぜ効くか / NG例 / 次回実践方法 を含める。
   - talkStyleModel、objectionPatterns、textDeliveryModel も可能な範囲で入れる。
   - 勝ち筋は、営業担当が読んで次回の会話を変えられる粒度で書く。単語の羅列ではなく、指導、理由、台詞、次回アクションを文章でつなぐ。
   - OpenAI Vector Store検索結果がある場合は、似た場面でトップ営業がどの順番で確認し、どこで不安を戻し、どの言葉で次アクションへ接続したかを抽象化して入れる。
7. 商談フェーズを特定する。
   - 冒頭・目的合わせ / 現状確認 / 課題深掘り / 提案 / 質疑応答 / 反論処理 / テストクロージング / クロージング / 次回合意 のどれに近いかを見る。
8. `stageStrategy` を作る。
   - `currentGoal`: このメールまたは次回接点で達成すべきこと
   - `keepUntilLater`: 今は言い切らない話題
   - `mustHearBeforeProposal`: 提案やプラン判断前に聞くこと
   - `planDecisionPath`: 判断材料を揃える順番
9. `customerSignals` と `temperature` を作る。
   - 温度感、懸念、購入動機、決裁観点、価格反応、家族相談、比較検討の有無を見る。
   - 温度感は高 / 中 / 低 と理由で書き、数値スコアは出さない。
10. `nextBestAction`、`hearingQuestions`、`recommendedAnswer` を作る。
   - 次の30秒または次回接点で営業が実行できる内容にする。
11. `benchmarkCoach` を作る。
   - トップ営業ならどう言うかを2〜6文の台本にする。
   - なぜ効くか、どの型を使ったか、文章量、説明順、資料URLやフォームへの接続を入れる。
12. `customerAttributePlaybooks` を作る。
   - 慎重・分析型、価格重視、成果重視、初心者、経験者、家族相談あり、即決寄り、比較検討中などから、今回使えそうなものを1〜2個だけ選ぶ。
13. 上記を、顧客送付用本文には混ぜず、社内確認用の商談フィードバック要素と4スタイル別全文メール案へ反映する。

この抽出結果は評価ログや残リスクとして出さない。営業が次に使う実践メモとして書く。

## 資料・検討結果入力フォームの固定表記

顧客送付用メールで案内する資料URLと検討結果入力フォームは、以下に統一する。
別の見出し、別URL、`〇〇〇` の仮置き、黄色マーカーは使わない。

```text
【個別相談会資料】
https://nextjp.co.jp/document/access/kobetsu_japan6.pdf

【検討結果入力フォーム】
https://docs.google.com/forms/d/e/1FAIpQLSd7K5Ki8TOdRVeE_zEZroSiDfqTFyS5RKf6HRXAa4_WWxg7iA/viewform
```

この2つは営業が差し替える箇所ではないため、黄色にしない。
`【】` 見出し数のチェックでは、この2つの定型ラベルは許容する。

## ネクストアクション

依頼文は途中で止めない。

NG:

- `状況だけ一度伺えればと。`
- `資料を見ていただけると。`
- `進められればと。`

OK:

- `状況だけ一度伺えますと助かります。`
- `お時間あるときにご覧いただけますと助かります。`
- `一つずつ確認させてください。`
- `こちらのメールにご返信いただいてもよろしいでしょうか。`

## 黄色箇所

### 記法（最重要・必ず守る）

黄色マーカーは**必ず `[黄色]` と `[/黄色]` で囲む**。後処理（Google Docs変換時）にこのタグの中身を実際の黄色ハイライトに変換するため、この形式が必須。
**`🟡` 絵文字・`【】`・「説明表だけで示す」などの独自記法は使わない**（変換されず黄色にならない）。

黄色にするもの:

- 冒頭の会話反映で、営業本人の温度感が必要な箇所 → `[黄色]…[/黄色]`
- **`ZOOM URL：〇〇〇` … 本文に出てくる全箇所（複数あれば全部）を必ず `[黄色]ZOOM URL：〇〇〇[/黄色]` で囲む**
  - 次回相談会URL・個別面談URL・次回ZOOM URL等、営業が送付前に差し替える必要があるZOOM系URLは1つ残らず黄色にする
  - （次回ZOOM予定が商談内に無い場合のみ、ZOOM URL自体を本文に入れない＝黄色も無し）
- 事実確認が必要な固有情報

黄色にしないもの:

- AIが商談から確実に抽出できる事実
- 動画URL
- 一般的な季語
- 定型の締め

## 禁止・注意表現

- `伝わってまいりました`
- `必ず成果が出ます`
- `安心して稼げます`
- `誰でも簡単に`
- 顧客が言っていない不安・感情の断定
- 営業担当の発話にない余白表現

## 数字・金額表記

金額は必ずアラビア数字で書く。漢数字で金額を書かない。

OK:

- `月々7万円ほど`
- `総額330万円ほど`
- `自己資金30万円ほど`
- `半年間のランニング費用が月々22,000円ほどから5,500円ほどになる`

NG:

- `月々七万円ほど`
- `総額三百三十万円ほど`
- `自己資金三十万円ほど`
- `二万二千円ほどから五千五百円ほど`

日数・時間・回数・割合も、原則としてアラビア数字で統一する。
ただし、自然な言い回しの `一度`、`一緒に`、`一つずつ` など金額・数量として扱わない語は漢字のままでよい。

## メール本文のフォーマット（最重要）

営業後送付メールは「**短い段落 + 必要最小限の箇条書き**」で書く。報告書フォーマット・社内文書フォーマットにしない。
読みにくさの主因は、商談背景、判断軸、作業内容、営業所感を1段落に詰め込むこと。情報の種類ごとに必ず分ける。

### 読みやすさの基準

- 1段落は原則1〜2文まで。長くても120〜160字程度に抑える。
- 1文に入れる論点は1つまで。`〜で、〜で、そして〜` と3論点以上をつながない。
- 挨拶、御礼、商談背景、判断軸、作業内容、参考動画、次アクションは別段落にする。
- 顧客の背景説明は厚くしすぎない。営業後メールでは「相手が読み返して判断しやすいこと」を優先する。
- `とても印象的でした`、`本気で考えていらっしゃるのが伝わってきました` のような営業所感は多用しない。使う場合も1文で止め、判断材料の説明へ進む。
- `八割九割`、`二つ` のような数量は、読ませたい情報なら `8〜9割`、`2つ` と書く。

### 禁止フォーマット

- 箇条書きだけで本文を構成しない。資料メモや議事録のように見える書き方は禁止。
- ただし、判断軸、作業内容、確認事項、次アクションのように読みやすさが上がる場合は、2〜4項目までの短い箇条書きを使ってよい。
  - 箇条書き1項目は1文まで。
  - 箇条書きの前後に、自然な導入文と締めの一文を置く。
  - 分析項目名や社内向けラベルは出さない。
- **見出し・Heading スタイルを使わない**（`#` `##` `■` `【見出し】` を本文構造として使わない）
  - 例外：件名・`【個別相談会資料】`、`【検討結果入力フォーム】`のような定型ラベル・黄色マーカー対象の `[黄色`〜`[/黄色]`はOK
- **宛名は冒頭1回のみ**。Title欄／Heading 1／本文中で繰り返さない
- **ドキュメントタイトル化しない**（「営業後送付フォロー」「個別面談フォローアップ」のような社内タイトルは付けない）

### 正しいフォーマット

- 件名 → 宛名 → 挨拶・名乗り → お礼 → 会話反映（1〜2文） → 判断軸の整理（短い箇条書き可） → 作業内容・資料案内（短い箇条書き可） → 参考動画（短文＋URL＋短文） → ネクストアクション（1つに絞る） → 安心導線（1文） → 季節の結び → 締め → 署名 → 固定の資料URL・検討結果入力フォーム
- 各ブロックは短く分ける。読点で長くつなぐより、改行で読みやすくする。
- 「3つの要点」は無理に1文へ詰め込まず、短い箇条書きで出してよい。

### 悪い例 → 良い例

NG（情報を1文に詰め込みすぎ）:

```
本日のお話の中で、判断軸として大きいのは、1日1〜2時間ほどの稼働で月15〜20万円規模を狙えるかどうか、半年ほどで支払いと売上のバランスが取れる見通しが立つかどうか、そして在庫を持たずに注文ベースで動かせる仕組みかどうか、というあたりだと受け取りました。
```

OK（短い導入文 + 判断軸の箇条書き）:

```
本日のお話では、特に下記の3点が判断軸になると受け取りました。

・1日1〜2時間ほどの作業を続けられるか
・月15〜20万円規模を目指すうえで、支払いと売上のバランスをどう見るか
・在庫を持たず、注文ベースで動かせる仕組みか

まずはこの3点を確認しながら、無理なく進められるかを見ていただければと思います。
```

NG（タイトル＋Heading化）:

```
営業後送付フォロー
井上由実子さま

# 営業後送付フォロー
井上由実子さま

お世話になっております。
```

OK（メール文として）:

```
井上由実子さま

お世話になっております。
株式会社NEXT（総合物販システム＿アクセス）の笠井でございます。
```

NG（宛名重複）:

```
井上由実子さま
〔件名行〕
井上由実子さま
お世話になっております。
```

OK（宛名1回）:

```
【件名】【株式会社NEXT】個別相談会の御礼(総合物販システム_アクセス)

井上由実子さま

お世話になっております。
```
