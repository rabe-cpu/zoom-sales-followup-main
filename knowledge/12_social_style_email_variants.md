# 12 Social Style Email Variants

## 目的

商談後メールの社内確認用ドキュメントに、4つのソーシャルスタイル別の全文メール案と営業フィードバックを出す。

このナレッジは、顧客のタイプを診断するためではない。
顧客の会話から「この人はDriver」などと判定せず、同じ商談事実を4つの伝え方に変換するために使う。

## 絶対ルール

- 顧客送付用メールは、原則として完成稿1通だけにする。
- 4スタイル別案は、社内確認用MD/docxにだけ入れる。
- 各スタイルのメール案は、差し替え段落ではなく **件名、宛名、本文、参考動画、ネクストアクション、署名、固定資料URL、固定フォームURLまで含む全文メール** にする。
- 4スタイルすべてで、顧客送付用完成稿と同じ商談事実、同じ固定URL、同じ参考動画URLを使う。スタイルごとに事実や送付物を変えない。
- 顧客の発話からソーシャルスタイルを断定しない。
- 「推定」「判定」「この顧客は〇〇型」のようなラベルを付けない。
- 商談で出ていない事実、感情、成果期待をスタイル別案に足さない。
- 成果保証、収益保証、過度な安心表現は禁止。
- 資料URLと検討結果入力フォームは `knowledge/04_email_generation_rules.md` の固定表記を使う。
- 営業後メール生成時は `knowledge/benchmark_playbooks/suzue_benchmark.md` を必ず参照し、4スタイル別の `benchmarkTalk`、`benchmarkCoach`、`winningPatterns` に反映する。
- 鈴江ベンチマークは営業型として使い、顧客が話していない事実を足す根拠にはしない。
- 音声・映像コーチング、声色・話速評価、録音練習、模写練習メニューは出力しない。
- 黄色マーカーは営業が編集・確認する箇所だけに使い、4スタイル案そのものを黄色にしない。

## 出力先

社内確認用ドキュメントに、以下の見出しで追加する。

```text
## ソーシャルスタイル別全文メール案（社内確認用）
```

このセクションは、営業担当が最終送信前に表現を選ぶための参考資料であり、顧客へそのまま送るものではない。

## 4スタイルの使い分け

### Driver / Driving

短く、結論先行で、次の行動が明確な文面にする。

- 重視するもの: 結果、スピード、行動、判断のしやすさ
- メールの寄せ方: 冒頭を短めにし、商談で確認した判断軸と次アクションを早めに出す
- 効きやすい表現: `判断材料としては、まず〇〇と〇〇をご確認いただければと思います。`
- 避ける表現: 長い背景説明、情緒的な前置き、結論のない検討依頼
- 次回アクション: 返信・確認・日程調整などを具体的に1つ置く

### Analytical

根拠、条件、数字、例外、リスク整理を厚めにする。

- 重視するもの: 正確性、根拠、条件整理、比較、リスク把握
- メールの寄せ方: 費用、作業時間、サポート範囲、回収イメージなどを会話事実の範囲で整理する
- 効きやすい表現: `判断材料としては、費用感、作業量、サポート範囲の3点を分けて見ていただくのが良いかと思います。`
- 避ける表現: `大丈夫です` の断定、根拠のない安心訴求、勢いだけの事例紹介
- 次回アクション: 不明点を洗い出し、追加確認の返信導線を置く

### Amiable

安心感、相談しやすさ、家族や周囲への説明しやすさを重視する。

- 重視するもの: 安心、安全、関係性、伴走、無理なく続けられること
- メールの寄せ方: 顧客の不安を否定せず、相談できる導線と確認手順を丁寧に置く
- 効きやすい表現: `ご家族にも説明しやすいように、費用面とサポート面は一つずつ整理していただければと思います。`
- 避ける表現: 急かす表現、不安を軽く扱う表現、強いクロージング
- 次回アクション: `気になる点だけでもご返信ください` のように心理的負担を下げる

### Expressive

将来像、前向きなイメージ、成功事例との接続を重視する。

- 重視するもの: 未来像、可能性、納得感、前向きな変化、ストーリー
- メールの寄せ方: 商談で出た目的や理想状態に接続し、参考動画や事例を見る意味を伝える
- 効きやすい表現: `今回のお話は、単に副業を増やすというより、今後の選択肢を広げるための判断材料になるかと思います。`
- 避ける表現: 条件説明だけで終わる、数字だけで押す、将来像を顧客以上に盛る
- 次回アクション: 参考動画を見た感想や、理想状態に近いかどうかを聞く

## 社内確認用に必ず含める項目

各スタイルについて、以下を出す。

```text
### Driver / Driving 向け

#### 全文メール案
（件名、宛名、本文、参考動画、ネクストアクション、署名、固定資料URL、固定フォームURLまで含む。差し替え段落だけで終わらせない）

#### この文面が効く理由
（decisionLogic: どの判断軸に刺さる設計か）

#### 顧客反応シグナル
（cues: 商談内で見えた発言・反応・質問。ただし顧客タイプの断定には使わない）

#### 営業担当が選ぶ目安
（どのようなお客様・商談状況ならこの案を使いやすいか。ただし顧客タイプの断定はしない）

#### 次回商談での質問例
（effectiveQuestions: 次に聞くと判断が進みやすい質問）

#### そのまま使える返答例
（effectiveReplies / recommendedAnswer: 営業担当が次回商談や返信でそのまま使える2〜6文の自然な台詞）

#### 価格・費用質問への返し方
（priceQuestionHandling: 金額だけで終わらせず、条件・目的・判断軸に戻す返し方）

#### 避ける言い方
（avoidedTalk: 失注や不信につながりやすい表現）

#### 伝え方メモ
（delivery: メール文量、説明順、資料URLやフォームへの接続、次アクションの明確さ）

#### 次の一手
（nextBestAction: 営業担当が送信後または次回接点で取る具体行動）

#### ベンチマーク営業トーク
（benchmarkTalk / benchmarkCoach: トップ営業ならどう言うか。商談事実と商品資料に根拠がある範囲だけで、2〜6文の台詞にする）

#### 文脈接続メモ
（contextBridge: 商談のどの発言・不安・判断軸から、この文面につなげているか）
```

4スタイルすべてで同じ構成にする。

## メール案の粒度

社内確認用では、4スタイルすべてを **全文メール案** として出す。

差し替え段落案だけで終わらせない。各全文メール案には、少なくとも以下を含める。

1. 件名
2. 宛名
3. 挨拶・名乗り
4. 商談で実際に話した内容の回収
5. 判断軸整理
6. 資料・シミュレーション案内
7. 参考動画の自然な案内文
8. 実際のYouTube URL
9. URL後の見る観点
10. ネクストアクション
11. 安心導線
12. 季節の結び
13. 締め・署名
14. `【個別相談会資料】` の固定URL
15. `【検討結果入力フォーム】` の固定URL

全文メール案は、顧客送付用完成稿と同じく段落形式で書く。本文中に営業分析、スタイル解説、社内確認情報を混ぜない。参考動画の選定理由は独立セクション化せず、メール本文中の自然な案内文にとどめる。スタイルごとの営業フィードバックは、全文メール案の後ろに別見出しで置く。

## 顧客送付用完成稿との関係

顧客送付用完成稿は、4スタイルのうちどれか1つに寄せ切らない。
商談事実、営業担当の実発話、顧客の判断軸を優先した標準案にする。

社内確認用では、標準案に加えて4スタイル別の全文案を出す。
営業担当は、送信前に社内確認用を見て、どの全文案をベースにするか、またはどの表現だけ標準案へ反映するかを選ぶ。

## 商談フィードバック要素の反映

`sales-analysis-app-openai-next` 側の商談フィードバック思想を、社内確認用に必ず反映する。

4スタイル別全文メール案を作る前に、まず以下の **商談フィードバック抽出** を内部で作る。
この抽出結果は顧客送付用本文には入れない。社内確認用では「商談フィードバック要素」として、営業担当が次回商談や返信で使える形に整えて出す。

```text
## 商談フィードバック要素

### 総合概要
- overallSummary: 商談の全体要約。顧客の状況、検討背景、判断軸、現在の進み具合を300〜500文字程度でまとめる
- meetingOutcome: won / lost / in_progress のような内部状態ではなく、自然な日本語で「検討中」「次回確認待ち」「前向きに比較中」などと書く
- benchmarkGapAnalysis.keyDifference: トップ営業・勝ち商談と比べた決定的な差分
- benchmarkGapAnalysis.missingActions: 不足行動を、フェーズ / 不足行動 / 影響 の形で書く。数値スコアは出さない

### 顧客インサイト
- hiddenNeeds: 顧客が明言していないが商談から見える本音・潜在ニーズ
- unspokenIssues: 顧客が言い切っていない不安、制約、迷い
- competingServices: 比較対象、代替案、迷っている選択肢。明確に出ていなければ「明確な競合名は未確認」と書く
- decisionCriteria: 顧客が何で判断しそうか。費用、作業時間、家族説明、回収イメージ、サポート範囲など
- buyingTemperature: 高 / 中 / 低 と理由。数値スコアは出さない

### 認知バイアス
- cognitiveBias: name / description / counterMeasure の形で、顧客または営業側に起きていそうな認知の偏りを書く
- 例: 現状維持バイアス、損失回避、確証バイアス、過度な楽観、情報過多による先送り

### 期待値のズレ
- expectationGap: topic / gap / solution の形で、顧客期待と実際の説明・条件のズレを整理する
- 費用、作業時間、回収時期、サポート範囲、家族説明、審査・契約条件などを優先する

### 良かった点
- strengths: item / detail の形で、営業担当が再現すべき良かった動きを3〜5個出す
- 単なる褒めではなく、どの発言・進め方が顧客の判断に効いたかを書く

### 改善ポイント
- improvementPoints: item / detail の形で、次回直すべき点を3〜5個出す
- 修正後の聞き方・言い方が分かるようにする

### AIコーチングカード
- coachingCards: 4枚を基本にする
- 各カードは theme / scene / insight / issue / strategy / script / outcome を必ず含める
- script は営業担当がそのまま言える2〜4文にする
- raw transcript、タイムスタンプ、評価点、評価ログは入れない

### 再現する勝ち筋
- winningPatterns: 勝ち筋名 / 使う場面 / 顧客シグナル / トップ営業の動き / コピートーク / なぜ効くか / NG例 / 次回実践方法 を含める
- talkStyleModel: 再現すべき営業像、語尾・敬語・柔らかさ、質問順、つなぎ台詞、クロージングのリズム
- textDeliveryModel: メール文量、説明順、質問の置き方、資料URLやフォームへの接続
- objectionPatterns: 反論・不安 / 隠れた懸念 / ベンチマーク返答 / 深掘り質問 / NG返答

### 商談フェーズ
- currentPhase: 冒頭・目的合わせ / 現状確認 / 課題深掘り / 提案 / 質疑応答 / 反論処理 / テストクロージング / クロージング / 次回合意 のどれに近いか
- currentGoal: このメールまたは次回接点で達成すべきこと
- keepUntilLater: 今は言い切らず、次回以降に回す話題
- mustHearBeforeProposal: 提案・プラン判断前に必ず聞くこと
- planDecisionPath: どの順番で判断材料を揃えるか

### 顧客シグナル
- customerSignals: 温度感、懸念、購入動機、決裁観点、家族相談、比較検討、価格反応など
- temperature: 高 / 中 / 低 と、その理由。ただし数値スコアは出さない

### 次の一手
- nextBestAction: 送信後または次回接点で営業担当が取る行動
- hearingQuestions: 次に聞くべき質問を優先順で最大3つ
- recommendedAnswer: 顧客から返信・質問が来たときにそのまま使える自然な返答

### ベンチマーク営業再現
- benchmarkTalk: トップ営業なら最初に置く短い一言
- benchmarkCoach.script: トップ営業がそのまま話す2〜6文の台本
- benchmarkCoach.whyItWorks: なぜ効くか
- benchmarkCoach.benchmarkPattern: 使った型
- benchmarkCoach.delivery: メール文量、説明順、資料URLやフォームへの接続、次アクションの明確さ

### 文脈接続
- contextBridge.sourceMoment: 商談内のどの発言・不安・判断軸を根拠にしているか
- contextBridge.insight: その文脈が今効く理由
- contextBridge.recommendedTalk: 営業担当がそのまま使える接続トーク

### 属性別対応
- customerAttributePlaybooks: 慎重・分析型、価格重視、成果重視、初心者、経験者、家族相談あり、即決寄り、比較検討中など、今回使えそうな属性を1〜2個だけ選ぶ
- effectiveQuestions: その属性に効く質問
- effectiveReplies: その属性に効く返答
- avoidedTalk: その属性で避ける言い方
- delivery: 伝え方
```

各スタイルの営業フィードバックには、上記の抽出結果を踏まえ、可能な範囲で以下を入れる。

- `cues`: 商談内で見えた顧客反応シグナル
- `decisionLogic`: そのスタイルで何を判断材料にしやすいか
- `effectiveQuestions`: 次回商談や返信で聞くと判断が進む質問
- `effectiveReplies`: 営業担当がそのまま使える返答例
- `priceQuestionHandling`: 商談初期や返信で価格・費用を聞かれた時の返し方
- `avoidedTalk`: 避ける言い方、NG表現
- `delivery`: メール文量、説明順、資料URLやフォームへの接続、メールの温度感
- `nextBestAction`: 送信後に営業担当が取る次の一手
- `benchmarkTalk`: トップ営業ならどう言うかの台詞
- `benchmarkCoach`: トップ営業の台本、効く理由、型、伝え方
- `stageStrategy`: 今のフェーズで達成すべきこと、後回しにする話題、提案前に聞くこと、プラン判断の道筋
- `contextBridge`: 商談内のどの話題・懸念・判断軸からこの文面へ接続したか
- `customerSignals`: 温度感、懸念、購入動機、決裁観点のシグナル
- `customerAttributePlaybooks`: 今回使えそうな顧客属性別の質問・返答・避ける言い方
- `overallSummary`: 総合概要
- `customerInsights`: 顧客インサイト
- `cognitiveBias`: 認知バイアス
- `expectationGap`: 期待値のズレ
- `strengths`: 良かった点
- `improvementPoints`: 改善ポイント
- `coachingCards`: AIコーチングカード
- `winningPatterns`: 再現する勝ち筋
- `riskAlerts`: 成果保証、価格断定、信販・審査・個人情報などで避けるべき注意点

価格・費用質問への返し方では、価格だけを即答して終わらせない。価格は隠さないが、目的、作業時間、予算感、導入時期、意思決定者、不安の種類を確認し、合うプランや判断条件へ戻す。

商談フィードバック要素は社内確認用の営業メモであり、顧客送付用本文には入れない。
評価ログや残リスクとしては出さない。あくまで営業が次に使うための実践メモとして書く。

## 評価観点

評価エージェントは、4スタイル別案について以下を確認する。

- 顧客タイプを断定していないか
- 商談にない事実を足していないか
- 4スタイルすべてで全文メール案になっているか。差し替え段落だけなら不合格
- 4スタイルの違いが、語尾だけではなく判断軸・情報量・次アクション・価格質問対応・伝え方メモに出ているか
- 商談フィードバック要素（overallSummary / customerInsights / cognitiveBias / expectationGap / strengths / improvementPoints / coachingCards / winningPatterns / stageStrategy / customerSignals / temperature / cues / decisionLogic / effectiveQuestions / effectiveReplies / priceQuestionHandling / avoidedTalk / delivery / nextBestAction / benchmarkTalk / benchmarkCoach / contextBridge / customerAttributePlaybooks）が営業メモとして反映されているか
- 顧客送付用本文に社内向けのスタイル分析が混ざっていないか
- Driverは短く具体的か
- Analyticalは根拠と条件が整理されているか
- Amiableは安心導線があり、急かしていないか
- Expressiveは未来像に接続しているが、顧客の発言以上に盛っていないか
