営業フォローメール自動生成システム 保守マニュアル

構成、認証、依存関係、障害切り分け、品質ルール更新のための保守手順

1. システム構成

このリポジトリは、Pythonスクリプトによる外部API連携と、Claude CodeのProject Skills / Agents / Hooksによるメール品質制御で構成されています。 Routineはポーリング方式でZoom録画を取りに行き、Webhookは使いません。 定期起動はcron-job.orgからClaude RoutineのAPI triggerを呼び出す方式です。Claude Routine側のSchedule triggerは重複防止のため使わず、API triggerだけを有効にします。GitHub Actionsは手動実行用のバックアップとして残します。

Chatwork担当者別ルーム通知

CHATWORK_ROOM_ID はデフォルト通知先です。追加担当者だけ別ルームへ通知したい場合は、Zoom商談のホストメールアドレスをキーにして CHATWORK_ROOM_ID_BY_HOST を追加します。

CHATWORK_ROOM_ID=100000000
CHATWORK_ROOM_ID_BY_HOST={"new1@example.com":"111111111","new2@example.com":"222222222"}

上記の場合、new1@example.com の商談は 111111111、new2@example.com の商談は 222222222 に通知されます。対応表にない担当者、エラー通知、全体警告は CHATWORK_ROOM_ID に通知されます。0件通知は CHATWORK_ROOM_ID に加えて、担当者別ルーム設定に含まれる各ルームにも通知されます。0件通知だけ必ず追加送信したいルームは CHATWORK_EMPTY_ROOM_IDS=437933333,424053694 のように指定します。

メールアドレスで判定できない場合のみ、担当者名をキーにした CHATWORK_ROOM_ID_BY_SALES_PERSON を使います。メンション先を変える場合は、投稿先ルームとは別に CHATWORK_ACCOUNT_ID_BY_HOST または CHATWORK_ACCOUNT_ID_BY_SALES_PERSON を設定します。

5. API連携の処理順

  1. cron-job.orgが、09:07から22:07まで30分間隔でClaude Routine API triggerを呼びます。
  2. Claude Routineは最初に python -m pip install -r scripts/requirements.txt を実行し、Google APIクライアントなどを準備します。
  3. scripts/routine_lock.py acquire がGoogle Drive上に routine_running.lock.json を作成します。既に新しいロックがある場合は、前回実行中として本処理へ進みません。
  4. ZoomClient.list_users() でアカウント内の有効ユーザーを取得します。
  5. ZoomClient.list_user_recordings() でユーザーごとの録画を取得します。
  6. fetch_vtt.py が対象外トピック、10分未満、VTTなし、処理済みを除外し、VTTを営業担当別Driveフォルダへ保存します。
  7. Claude Routineがメールを生成し、顧客送付用MDと社内確認用MDを作ります。
  8. GoogleDriveClient.upload_md_as_doc() がGoogle Docsへ変換アップロードします。
  9. GoogleDriveClient.apply_yellow_highlights() が [黄色]...[/黄色] をDocs上の黄色背景に変換します。
  10. GmailDraftClient.create_draft() が下書きを作成します。
  11. Drive保存とGmail下書きが成功した後、add_to_ledger() で台帳を更新します。
  12. notify_chatwork.py が処理結果を通知します。

6. 営業担当別VTT保存設定

VTT保存先は、メールアドレス単位の VTT_DRIVE_FOLDER_BY_HOST を最優先し、次にZoomユーザー名と担当名の一致で VTT_DRIVE_FOLDER_BY_SALES_PERSON を使います。 どちらも一致しない場合は未割当フォルダへ保存します。

7. 台帳、ロック、冪等性

重複処理防止はGoogle Drive上の processed_meetings.json で行います。 meeting_id が台帳に登録済みなら、次回以降は処理対象から外れます。 30分間隔起動では、これに加えて routine_running.lock.json を使い、前回Routineがまだ実行中の場合に新しい実行を止めます。

  • processed_meetings.json: 商談単位の再処理防止。Drive保存とGmail下書きが成功した後に更新します。
  • routine_running.lock.json: Routine実行単位の重複防止。開始時に作成し、Chatwork通知後の --release-lock で削除します。
  • ROUTINE_LOCK_TTL_MINUTES: ロックの事故残りを破棄するまでの分数です。未指定時は90分です。
台帳を削除すると過去録画が再処理されます。再処理したくない場合は台帳を消さず、必要に応じて対象meeting_idだけを確認してください。

10. 障害切り分け

症状原因候補確認箇所
Zoom認証エラー Account ID、Client ID、Client Secret不一致。S2S App未Activate。 Zoom Marketplace / Routine環境変数 / zoom_client.py
ユーザー一覧取得エラー user:read:admin 相当の権限不足。 Zoom S2S OAuth AppのScopesと管理者ロール。
録画が取れない recording read系スコープ不足、対象ホストがLicensedでない。 Zoom録画設定、対象ユーザー、S2S Scope。
VTTがない Audio transcriptがOFF、録画処理中、ローカル録画。 Zoom Recording & Transcript設定。
VTTが未割当に保存される Zoomユーザー名が担当名と一致しない、host_email対応表が未設定。 VTT_DRIVE_FOLDER_BY_HOST、Zoomユーザー名、Step1 JSONの host_name。
Drive保存失敗 refresh token無効、Drive folder ID誤り、権限不足。 gdrive_client.py、Google OAuth、保存先フォルダ。
黄色ハイライト失敗 Docs scope不足、タグ不整合、Docs APIエラー。 apply_yellow_highlights()、顧客/社内MDのタグ。
Gmail下書き失敗 Domain-wide delegation未設定、Gmail API無効、host_email不一致。 gmail_client.py、Google Workspace Admin、サービスアカウント。
Chatwork通知失敗 API token無効、Room ID誤り、投稿権限なし。0件通知の場合も notify_chatwork.py の標準出力に Chatwork通知成功 room_id=... が出ていなければ未送信として扱う。 notify_chatwork.py、Chatworkルーム、Routine実行ログ。
Routineだけ失敗 環境変数不足、ネットワーク許可不足、依存関係未インストール、API trigger token不一致。 Routine Run log、Setup command、Environment、GitHub Actions Secrets。
30分ごとの起動が動かない cron-job.orgのjob停止、Schedule設定不備、Time zone不一致、Headers不備、Claude Routine API Token無効。 cron-job.orgのHistory/Test run、Fire URL、Authorization、anthropic-version、anthropic-beta。
前回実行中として止まる 正常な重複防止、または routine_running.lock.json の事故残り。 Drive上のロックファイル、Routineログの status: locked、ROUTINE_LOCK_TTL_MINUTES。

11. ローカル確認コマンド

認証情報が手元にある場合、Step単位で切り分けできます。

# 依存関係
python -m pip install -r scripts/requirements.txt

# Routine重複防止ロックの状態確認
cd scripts
python routine_lock.py status

# Zoomから未処理VTT取得
cd scripts
python fetch_vtt.py

# Chatwork疎通
python notify_chatwork.py --empty

# Drive/Gmail保存は、生成済みMDを指定して実行
python save_to_gdrive.py \
  --customer "山田太郎様" \
  --send-date "2026-06-01" \
  --meeting-id "test-meeting-id" \
  --customer-md "/tmp/output/山田太郎様/01_山田太郎様_顧客送付用.md" \
  --internal-md "/tmp/output/山田太郎様/01_山田太郎様_社内確認用.md" \
  --gmail-user "sales@example.com"
テスト用の meeting-id を本番台帳に登録すると、そのIDは処理済み扱いになります。本番Driveを使うテストでは命名と台帳影響を明確にしてください。

12. 変更時のリリース手順

  1. 変更対象を確認し、関連するSkill、Agent、Knowledge、Routine Promptの整合を取ります。
  2. シークレットや一時出力が差分に含まれていないか確認します。
  3. 可能ならローカルで該当スクリプトを単体確認します。
  4. Claude RoutineのPromptを変更した場合は、Routine画面側にも反映します。
  5. cron-job.orgやAPI triggerを変更した場合は、cron-job.orgのHeadersとClaude RoutineのAPI trigger設定を確認します。
  6. GitHub Actionsの手動バックアップを変更した場合は、GitHub Secretsも確認します。
  7. Run nowで1件だけ検証します。
  8. Drive保存、Gmail下書き、Chatwork通知、台帳更新を確認します。
  9. 初回運用分は人間が出力Docsを目視確認します。

13. 切り戻しと停止

  1. cron-job.orgのjobをPauseまたはDisableします。
  2. 必要に応じてGitHub Actionsの手動workflowを無効化します。
  3. Claude RoutineのAPI triggerを削除またはTokenを無効化します。
  4. Claude Routine側にSchedule triggerが残っていないか確認します。
  5. 必要に応じてRoutine環境変数からシークレットを削除します。
  6. Zoom S2S OAuth AppをDeactivateします。
  7. Google OAuth refresh tokenを無効化します。
  8. Chatwork API tokenを無効化または再発行します。
  9. Drive出力フォルダの共有範囲を見直します。
台帳を削除すると再処理が発生します。停止や切り戻しで台帳を触る場合は、削除ではなくバックアップを取ってから編集してください。

14. 保守チェックリスト

週次
  • Chatwork通知に失敗がないか確認。
  • Driveに重複Docsが増えていないか確認。
  • Gmail下書きが営業担当に作成されているか確認。
月次
  • Zoom、Google、Chatworkの認証状態を確認。
  • 動画カタログと営業担当ナレッジを見直す。
  • RoutineのRun logに依存関係警告がないか確認。
変更時
  • SkillとKnowledgeの重複・矛盾を確認。
  • Stop Hookの要求証跡が満たせるか確認。
  • Run nowで小さく検証してから本番化。
事故時
  • cron-job.orgとClaude Routine API triggerを止める。
  • 送信済みメールの有無を確認。
  • シークレット漏えいの可能性があれば即時ローテーション。
