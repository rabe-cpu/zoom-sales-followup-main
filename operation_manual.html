営業フォローメール自動生成システム 運用マニュアル

Zoom録画の文字起こし取得から、メール生成、Drive保存、Gmail下書き、Chatwork通知までの日常運用手順

1. このシステムで行うこと

このシステムは、Zoomクラウド録画のVTT文字起こしをもとに、商談後の営業フォローメールを自動生成します。 Routineはメール送信までは行いません。顧客送付用Google Docs、社内確認用Google Docs、営業担当Gmailの下書き、Chatwork通知を作成し、最後の確認と送信は人間が行います。 現在の定期起動は、cron-job.orgからClaude RoutineのAPI triggerを呼ぶ方式です。Claude Routine側のSchedule triggerは使わず、API triggerだけを有効にします。GitHub Actionsは手動実行用のバックアップとして残します。

自動送信は禁止です。営業担当または確認担当者が、黄色箇所、次回ZOOM URL、宛先、顧客固有情報を確認してから送信します。

2. 日常運用の担当範囲

営業担当

Gmail下書き、顧客送付用Docs、黄色箇所、ZOOM URL、宛先を確認し、問題なければ手動送信します。

運用担当

Routineの実行結果、Chatwork通知、Drive保存、台帳更新を確認します。失敗時は一次切り分けを行います。

保守担当

認証情報、依存関係、Zoom/Google/Chatwork連携、SkillやKnowledgeの更新を管理します。

3. 通常運用フロー

  1. 商談をZoomクラウド録画し、Audio transcriptが生成されるのを待ちます。
  2. cron-job.orgが9時台から22時台まで、09:07、09:37、10:07...22:07の30分間隔でClaude RoutineのAPI triggerを呼び出します。
  3. Claude Routineが起動したら、最初にPython依存関係をインストールします。
  4. Google Drive上の重複防止ロック routine_running.lock.json を取得します。前回実行中なら何も処理せず終了します。
  5. scripts/fetch_vtt.py が未処理のVTTだけを取得し、営業担当別Google Driveフォルダへ保存します。
  6. 商談ごとに、顧客送付用メールと社内確認用メールが生成されます。
  7. scripts/save_to_gdrive.py がGoogle DriveへDocs保存し、Gmail下書きを作ります。
  8. 保存と下書き作成が成功した商談だけ、処理済み台帳に登録されます。
  9. scripts/notify_chatwork.py がChatworkへ結果を通知します。担当者別ルーム設定がある場合は、商談の host_email または担当者名に応じて別ルームへ分けて通知します。
  10. 営業担当が下書きを確認し、必要箇所を修正して手動送信します。

4. Run nowで手動実行する場合

定期実行を待たずに処理したい場合は、Claude Routineの画面からRun nowを実行するか、GitHub Actionsの Scheduled Claude Routine から Run workflow を実行します。cron-job.orgの定期実行と同時に押すと二重起動になるため、手動実行は片方だけにします。

  1. Claude Code on the webで対象Routineを開きます。
  2. Run now を押します。
  3. GitHub Actionsから試す場合は、GitHubのActionsタブで Scheduled Claude Routine を選び、Run workflow を押します。
  4. Routineログ冒頭で python -m pip install -r scripts/requirements.txt と python scripts/routine_lock.py acquire が実行されることを確認します。
  5. 実行ログでStep1からStep4まで進んだか確認します。
  6. Chatwork通知が届いたか確認します。担当者別ルーム運用の場合は、既存担当はデフォルトルーム、新担当者は設定した個別ルームに届いているか確認します。
  7. Driveに顧客別フォルダと2つのDocsができたか確認します。
緑の実行ステータスだけでは成功判断しません。ロック取得、Drive保存、Gmail下書き、台帳更新、Chatwork通知、最後の --release-lock によるロック解除まで確認してください。

5. 出力物の確認方法

出力物確認すること問題がある場合
文字起こしVTT Chatwork通知またはStep1 JSONの vtt_drive_url から開けること。担当者別フォルダに入っていること。 未割当フォルダに入っている場合、Zoomホスト名または環境変数の対応表を確認します。
顧客送付用Docs 営業口調抽出、残リスク、黄色タグ文字などの社内確認情報が混ざっていないこと。 送信せず、社内確認用DocsとRoutineログを確認します。
社内確認用Docs 黄色箇所、ソーシャルスタイル別全文案、商談フィードバック要素、最終確認が見えること。評価ログと残リスクは出さない。 不足していれば人間確認を優先し、保守担当へ連絡します。
Gmail下書き 営業担当本人のGmailに作成されていること。Toは空欄でも可。 宛先、件名、本文、URLを送信前に修正します。
Chatwork通知 顧客送付用URL、社内確認用URL、Gmail下書きID、残リスクがあること。担当者別ルーム運用の場合は、該当担当のルームへ届いていること。 通知がなければRoutineログ、Chatwork認証、CHATWORK_ROOM_ID_BY_HOST のメールアドレスとルームIDを確認します。

担当者別ルーム通知は、商談の host_email が CHATWORK_ROOM_ID_BY_HOST に一致した場合に分岐します。対応表にない担当者、エラー通知はデフォルトの CHATWORK_ROOM_ID に届きます。0件通知はデフォルトルームに加えて、担当者別ルーム設定に含まれる各ルームにも届きます。

6. 営業担当の送信前チェック

  • 宛先メールアドレスが正しい。
  • 顧客名、敬称、会社名、商談内容に誤りがない。
  • 黄色箇所を確認し、必要に応じて自然な表現へ修正した。
  • ZOOM URL：〇〇〇 が残っている場合、正しいURLへ差し替えた。
  • 参考動画URLが実URLになっている。
  • 成果保証、収益保証、過度な断定表現がない。
  • 残リスクや営業口調抽出など社内情報が本文に混ざっていない。
[黄色]、[/黄色]、参考動画URL：〇〇〇、社内確認情報、残リスク が顧客送付本文に残っている場合は送信しないでください。

7. 0件通知の場合

Chatworkに「処理対象の商談はありませんでした」と通知された場合、異常とは限りません。次の条件を確認してください。

  • 本日JSTの対象商談がなかった。
  • Zoom録画はあるが、VTT文字起こしがまだ生成されていない。
  • 商談直後で、クラウド録画やAudio transcriptの処理がまだ終わっていない。
  • 商談時間が10分未満だった。
  • 対象ホストが ALLOWED_HOST_EMAILS に含まれていない。
  • すでに processed_meetings.json に登録済みだった。
9:00に商談が始まった場合、09:07や09:37のRoutineではまだ拾えないことがあります。Zoom録画が完了し、Audio transcript / VTT が生成された後の次回実行で取得されます。

9. 失敗時の一次対応

症状運用担当が確認すること次の対応
Chatworkにエラー通知が来た 通知本文のStep番号とエラー内容。 該当Stepを保守担当へ共有します。
DriveにDocsがない RoutineログでStep3まで進んだか。 Google認証、DriveフォルダID、保存権限を確認します。
Gmail下書きがない Chatwork通知の gmail_draft_id と gmail_user。 営業担当Gmail、Domain-wide delegation、host_emailを確認します。
同じ商談が再処理される Step3が最後まで成功し、台帳更新されたか。 台帳更新前に失敗している可能性があります。重複Docsを整理し、原因を保守担当へ連絡します。
メール品質が低い 社内確認用Docsの営業チェック欄、ソーシャルスタイル別全文案、商談フィードバック要素。 人間が修正して送信し、必要ならKnowledge更新を依頼します。

10. 手動フォールバック

Routineが不調な場合は、ローカルまたはClaude Code上で同じ流れを手動実行します。詳細は routine/MANUAL_FALLBACK.md を参照してください。

cd scripts
python fetch_vtt.py

# 取得された vtt_path をClaude Codeに渡してメール生成

python scripts/save_to_gdrive.py \
  --customer "{顧客名}" \
  --send-date "{送付日}" \
  --meeting-id "{uuid}" \
  --customer-md "/tmp/output/{顧客名}/01_{顧客名}_顧客送付用.md" \
  --internal-md "/tmp/output/{顧客名}/01_{顧客名}_社内確認用.md" \
  --gmail-user "{Zoomのhost_email}"
手動実行でも、保存とGmail下書き作成が成功した後に台帳へ登録されるため、基本的には重複処理を避けられます。30分間隔運用では、台帳に加えて routine_running.lock.json により「前回実行中なら新しい実行を止める」制御も行います。
